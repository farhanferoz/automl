## A2 — the selection-data fraction

**No LIBRARY method has an internal held-out fraction at all.** `fit_router` (all three classes) and `held_out_arbiter_advantage` (ProbReg) all take `x_val`/`y_val` (or `x`/`y`) as REQUIRED caller-supplied arguments — none of them call `train_test_split` or any splitting utility internally, none has a `val_fraction`/`test_size` parameter. The "selection-data fraction" question only has an answer at the SCRIPT layer, and scripts disagree:
- `automl_package/examples/report_a_benchmark.py:334` (ProbReg k-select) and `:376` (general per-cell split) both hardcode `test_size=0.3` via `sklearn.model_selection.train_test_split`. Same value in both places within this file.
- `automl_package/examples/moe_flexnn_comparison.py:89`: `VAL_FRAC = 0.2` module constant ("local train-pool -> (train, val) split fraction, both regression toys and A5's train half"), consumed by `_split_pool` (189-198) inside `build_data` (201-224) — this is the split BOTH `fit_router` (depth AND width) and `best_fixed_val_selected` (depth AND width cheap-global) draw their `x_val`/`y_val` from in this script.
- `automl_package/examples/capacity_ladder_f2.py` (depth expensive-sweep): does not use a fraction at all — `x_tr, y_tr = make_fn(n=cfg.n_train, seed=seed)` and `x_te, y_te = make_fn(n=cfg.n_test, seed=seed+500)` are INDEPENDENT generator draws with explicit sample counts, not a split of one pool.
- `tests/test_phase3_dynamic_k.py:208/220/266`: `test_size=0.3` for general fit/predict tests; the router-specific test (`test_distilled_router_matches_or_beats_best_global_fixed_k`, ~547-589) uses three INDEPENDENTLY DRAWN sets (`x_train`/`x_val`/`x_test`, seeds 0/1/2, sizes 600/300/600) rather than a fraction split of one pool, and its "best_global_nll" baseline is computed directly on `x_test`/`y_test` (a test-set oracle, not a held-out-val selection) — explicitly flagged in its own docstring as "the fixed-k baseline test-set-optimal information the router never sees — a conservative, favorable-to-the-baseline comparison," i.e. deliberately NOT apples-to-apples with a genuine val-selected global capacity.

**Answer: it is never a baked-in library constant; it is always a caller/script-level choice, and the two research scripts that do hardcode one disagree (0.3 vs 0.2).**

## A3 — the API shape and its silent failures

Exact signatures (all confirmed by direct grep across `automl_package/models/*.py`):
- `ProbabilisticRegressionModel.predict(self, x, filter_data=True, inference_mode: str = "soft")` (probabilistic_regression.py:598) and `.predict_uncertainty(self, x, filter_data=True, inference_mode: str = "soft")` (694) — BOTH take `inference_mode`, both validate `inference_mode not in ("soft", "routed")` and RAISE `ValueError` (613-614, 708-709). If `inference_mode="routed"` but no router fitted: RAISES `RuntimeError` (737, "No router fitted; call fit_router() before predict(...)"). Clean fail — no silent wrong-model output for THIS class.
- `FlexibleWidthNN.predict(self, x, filter_data=True, width: int|None=None, inference_mode: str = "fixed")` (flexible_width_network.py:192). NO `predict_uncertainty` override at all — inherits `PyTorchModelBase.predict_uncertainty(x, filter_data=True)` (base_pytorch.py:311), which has NO `inference_mode`/`width` concept whatsoever. Validates `inference_mode not in ("fixed", "routed")` → raises `ValueError` (204-205); `inference_mode="routed"` without a fitted router → raises `RuntimeError` (208-209). BUT: `predict_uncertainty`'s silent-failure risk is real — see the dedicated finding below (default `uncertainty_method=CONSTANT` is safe; explicit `PROBABILISTIC`/`MC_DROPOUT` is a shape-confusion bug, not blocked by any validation, uncovered by tests/test_flexible_width_network.py).
- `FlexibleHiddenLayersNN.predict(self, x, filter_data=True, inference_mode: str = "soft")` (flexible_neural_network.py:386) — THREE modes: "soft"/"hard"/"routed", raises `ValueError` on anything else (397-398), raises `RuntimeError` if "routed" without a fitted router (401-402). `.predict_uncertainty(self, x, filter_data=True)` (502) has NO `inference_mode` parameter AT ALL — always uses the full soft-selection forward pass (line 518). Consequence: a caller who calls `predict(x, inference_mode="routed")` then `predict_uncertainty(x)` gets a POINT ESTIMATE from the routed depth and an UNCERTAINTY from the soft-blended depth — two different depths silently paired into one result, with zero warning or error. This is the sharpest silent-mismatch finding in the audit: the failure mode is neither "raise" nor "warn" — it's "return a self-inconsistent (mean, std) pair with no signal anything is wrong."
- `IndependentWeightsFlexibleNN.predict(self, x, filter_data=True)` (independent_weights_flexible_neural_network.py:388) and `.predict_uncertainty(self, x, filter_data=True)` (438) — NEITHER has an `inference_mode` parameter; there is no router, no hard/soft choice to forget. "Forgetting the mode flag" is not applicable to this class because the flag doesn't exist — the caller gets the ONLY behavior available (per-sample hard indexing into `independent_networks[n_actual-1]`).

Enums vs raw strings (checked against automl_package/enums.py, 273 lines, read in full): `inference_mode` ("soft"/"hard"/"routed"/"fixed") is a raw string on EVERY class that has it — no `InferenceMode`/`PredictMode`/`SelectionMode` enum exists anywhere in enums.py. This is a closed set (2-3 values per class, validated by an explicit `if x not in (...)` tuple) that the repo's own CLAUDE.md hygiene rule ("closed sets of variants get a type, not strings") would flag — the pattern is duplicated three times (once per class) with three DIFFERENT closed sets (`{"soft","routed"}`, `{"fixed","routed"}`, `{"soft","hard","routed"}`), which is itself evidence for A5 (no single enum could serve all three today without a value only some families support). `k_prior_type` (probabilistic_regression.py:90, `"uniform"`/`"geometric"`) and `loss_type` (`"nll"`/`"beta_nll"`, per CLAUDE.md) are two further raw-string closed sets inside ProbReg not covered by any enum in enums.py.

## A4 — cost accounting

`docs/plans/capacity_programme/shared/metrics-accounting.md` (read in full) specifies Task S2 as "Analytic (not timed) accounting" (capacity_accounting.py:14 docstring) — the doc itself never promises wall-clock capture; only `param_count`/`executed_flops` (params/FLOPs), explicitly scoped as static/analytic. This is faithfully implemented: `automl_package/examples/capacity_accounting.py` (`functools.singledispatch`-based `executed_flops`) has registrations for exactly: `nwn.NestedWidthNet`, `nwn.SharedTrunkPerWidthHeadNet`, `nwn.IndependentWidthNet`, `nwn.SharedReadoutPerWidthAffineNet` (all from the research module `nested_width_net.py`, NOT the package), `FlexibleHiddenLayersNN.FlexibleNNModule` (depth family — the real package class), `FlexibleWidthNN.FlexibleWidthNNModule` (width family — the real package class), `DepthNetShapeDescriptor`, `MoEShapeDescriptor` (stand-ins for planned-but-never-built classes, per the module's own docstring, capacity_accounting.py:20-31). VERIFIED by `grep -n "@executed_flops.register" automl_package/examples/capacity_accounting.py` → exactly those 8 registrations, no more.

**Confirmed ABSENT**: no `executed_flops` registration for `ProbabilisticRegressionNet` (k family) anywhere — this matches `ProbabilisticRegressionModel.fit_router`'s own docstring, which states explicitly: "no `executed_flops` accounting exists yet for this architecture (capacity-programme scope)... `cost_fn=None` (default) leaves it disabled" (probabilistic_regression.py:791-794). **Confirmed ABSENT**: no registration for `IndependentWeightsFlexibleNN.IndependentWeightsFlexibleNNModule` either — that class has no cost accounting AND no router (see matrix), so `mean_deployed_cost` is unreachable for it by construction, not merely unwired.

Net: cost accounting IS wired end-to-end (`fit_router`'s `cost_fn` default → `capacity_accounting.executed_flops` → `DistilledCapacityRouter.mean_deployed_cost`) for exactly 2 of 4 model classes: `FlexibleWidthNN` and `FlexibleHiddenLayersNN`. `ProbabilisticRegressionModel.fit_router` accepts a `cost_fn` argument but wires no default (caller must supply one or `mean_deployed_cost` stays disabled). `IndependentWeightsFlexibleNN` has neither a router nor a cost function to attach one to. The doc (S2) is honestly scoped and does not overclaim — the gap is real but is a SCOPE gap the doc itself flags ("Depth nets... and MoE... have NO real class yet"), not a doc-vs-code discrepancy for the two classes it does claim to cover.

## A5 — what a unified API would have to cover (descriptive obstacles only)

1. **Precondition asymmetry**: ProbReg's per-input router AND its cheap-global arbiter both hard-require `n_classes_selection_method=NESTED` (raise `RuntimeError` otherwise, probabilistic_regression.py:805-809, 893-897) — a training-time choice gates an inference-time capability. Neither `FlexibleWidthNN.fit_router` nor `FlexibleHiddenLayersNN.fit_router` has an analogous training-time precondition; width/depth routing works regardless of `width_selection_method`/`layer_selection_method` at fit time (confirmed no such check in either file). A unified `predict(mode=...)` cannot promise "routed" works uniformly across families without surfacing this precondition difference.
2. **Bypass rung**: ProbReg's capacity grid includes a `(1,)` "bypass" (direct regression, no classification bottleneck) as a first-class rung (probabilistic_regression.py:788, `DIRECT_REGRESSION_K_SENTINEL`). Width and depth have no analogous "opt out of the dial entirely" rung — their grids are `1..max_hidden_layers` / `self.widths`, always going through SOME capacity value, never a qualitatively different bypass architecture.
3. **`predict()` argument shape differs per family**: ProbReg/`FlexibleHiddenLayersNN` key the fixed-value override off `inference_mode` alone (implicit "the trained default" when not routed); `FlexibleWidthNN` ADDS a separate `width: int | None` parameter alongside `inference_mode`. `IndependentWeightsFlexibleNN` has no override parameter at all. A single enum-at-construction, no-mode-flag-on-predict design has to decide whether the fixed-value override becomes a universal `predict(x, capacity=...)` param or disappears entirely for classes (like today's IndependentWeightsFlexibleNN) that never had one.
4. **Scoring metric per family differs and is baked into each class's own `fit_router`'s closed-over `eval_fn`**: ProbReg scores per-sample Gaussian NLL (probabilistic_regression.py:820-828, using `forward_at_k`'s (mean, log_var) output); width/depth score squared-error (regression) or 0/1 error (classification) (flexible_width_network.py:275-287, flexible_neural_network.py:476-489). `DistilledCapacityRouter.fit` itself is metric-agnostic (`eval_fn` is caller-supplied) — but a unified API surface would need to either expose this choice or hide three genuinely different loss semantics behind one call.
5. **The "cheap global" mechanism has NO shared implementation across families today** — it's LIBRARY (partial, per-input-shaped) for ProbReg, SCRIPT-only (and shared across width+depth via one function, `best_fixed_val_selected`) for width/depth, and does not exist at all for IndependentWeightsFlexibleNN. A unified API would need to either promote `best_fixed_val_selected`'s pattern into `DistilledCapacityRouter` (or a sibling) as a genuine LIBRARY primitive, or accept that "cheap global" stays permanently script-only for 3 of 4 classes.
6. **Grid/rung ordering convention**: `DistilledCapacityRouter`'s `_cheapest_within_tolerance_labels` (distilled_router.py:63-81) relies on `capacity_grid` being ordered CHEAPEST-FIRST (docstring: "argmax on a boolean row returns the FIRST True ... because capacity_grid columns are ordered cheapest-first BY THIS MODULE'S CONVENTION") — this is a caller-enforced invariant, not type- or runtime-checked anywhere in `DistilledCapacityRouter.fit` (confirmed: no sort/order assertion in the 232-line file). ProbReg's default grid `[(k,) for k in range(1, max+1)]` (probabilistic_regression.py:811) and width/depth's analogous defaults are all naturally ascending, but a unified constructor-time enum would need to either enforce this ordering or document it loudly, since nothing currently stops a caller from passing an unordered `capacity_grid` and silently getting wrong "cheapest-within-tolerance" labels.
7. **IndependentWeightsFlexibleNN is not merely "missing a mechanism," it is missing the underlying PRIMITIVE** (`forward_at_depth`/`forward_at_depths`) that every other family's router/cheap-global/sweep path is built on. A unified enum-at-construction API cannot uniformly offer `mode=ROUTED` for this class without first adding that primitive — it is not a wiring gap, it is a missing method on the `nn.Module`.

## MOST SERIOUS CODE-LEVEL GAPS (most serious first)

1. **`FlexibleHiddenLayersNN.predict_uncertainty` has no `inference_mode` parameter, so `predict(inference_mode="routed")` + `predict_uncertainty()` silently returns a (mean, std) pair computed at TWO DIFFERENT DEPTHS with no error, warning, or any signal** (flexible_neural_network.py:386 vs 502-531). This is the single most dangerous finding: it produces a self-inconsistent, silently-wrong scientific artifact (a UQ interval that doesn't correspond to the point estimate it's paired with) rather than a crash.
2. **`IndependentWeightsFlexibleNN` has zero of the three selection mechanisms wired at the class level** — no `fit_router`, no cheap-global helper, no `inference_mode` on `predict()`, and crucially no `forward_at_depth`/`forward_at_depths` primitive on its module at all, which is presumably WHY the other two were never added. It is asymmetric with its sibling `FlexibleHiddenLayersNN` in a way the CLAUDE.md docs ("Variant: IndependentWeightsFlexibleNN has separate weights per depth") do not surface.
3. **`FlexibleWidthNN.predict_uncertainty` (inherited, unmodified from `PyTorchModelBase`) will silently mis-index a stacked `(len(widths), N, output_size)` tensor as if it were `(N, output_size)` if a caller ever sets `uncertainty_method=PROBABILISTIC` or `MC_DROPOUT`** — nothing in `__init__` blocks this, and `tests/test_flexible_width_network.py` has zero coverage of `uncertainty_method`/`predict_uncertainty` at all (verified by direct grep). Currently latent (default `uncertainty_method=CONSTANT` avoids the broken path) but a live footgun for the first caller who deviates from the default.
4. **No `executed_flops`/cost accounting exists for ProbReg's k dial at all**, and `fit_router`'s `cost_fn` default is `None` (disabled). This fails LOUDLY when misused — `DistilledCapacityRouter.mean_deployed_cost` raises `RuntimeError` if `costs_` is `None` (distilled_router.py:229-230, verified) — so it is not a silent-corruption bug. But it means cross-family cost comparisons are currently wireable out-of-the-box for width+depth only; ProbReg is silently EXCLUDED from any report loop that assumes every family's router has a working `mean_deployed_cost`, unless the caller notices and supplies their own `cost_fn`.
5. **The "cheap global" cell has no LIBRARY implementation for ANY family that matches the literal "one dial value for the whole dataset, read off an already-trained model, no retraining" definition** — the only implementation that fits that description (`best_fixed_val_selected`, moe_flexnn_comparison.py:309-336) lives in a research script and covers width+depth only; ProbReg's designated reference (`held_out_arbiter_advantage`) is per-input by construction, and the code that WOULD make a matching global-k selector possible (`NestedStrategy.all_rung_log_likelihood`) has zero callers anywhere in the repo.
6. **Three different raw-string closed sets for `inference_mode` across three classes, plus `k_prior_type`/`loss_type` inside ProbReg**, none backed by an enum in `automl_package/enums.py` — violates the repo's own stated hygiene rule and is the concrete blocker A5 obstacle 3 names for any single unified mode enum.
