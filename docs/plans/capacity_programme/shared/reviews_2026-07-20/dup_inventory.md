# FlexNN duplication inventory — automl_package (read-only research pass)

Scope note up front: the team-lead brief listed 9 scripts as containing "a local router
implementation or a local `fit_router`." Verified: only **3 of the 9** actually define their own
router `nn.Module` class. The other 6 import and reuse one of those three, adding only new
training-objective / labeling-rule / eval-protocol code on top. This distinction matters a lot for
migration risk (see Deliverable 1) and is stated explicitly below rather than repeating the
brief's framing uncritically.

**Are they behaviourally equivalent? NO — several real differences, not just refactor-safe renames:**
- **Objective:** most variants train router via CE-to-a-label (hard or soft). `capacity_ladder_s2.py` trains the SAME architecture directly on the blend-NLL deployment metric (`_train_router_direct`) — this is a different optimization target, not just a different label source, and is not present at all in the package.
- **Eval protocol:** two genuinely different downstream uses coexist: (a) hard-argmax routing + MSE/0-1 error on the routed choice (sinc_width `_selector_eval_mse`/`_route_hardpick_mse`, depth_selection_toy `deploy_battery`, the package's `DistilledCapacityRouter.route`/`mean_deployed_cost`), vs (b) full-distribution probability-WEIGHTED blend-NLL (`_selector_eval`, t2's `_eval_router`/`_blend_nll`, s1/s2/h1's blend machinery) that never collapses to a single choice. The package only implements (a). It has no blend-NLL evaluation path at all.
- **Input shape:** scalar-only (k6, s1, s2, h1, sinc_width's underlying router) vs vector (t2, depth_selection_toy, joint_capacity_toy, package). Two independent vector ports exist (t2 and depth_selection_toy) that don't know about each other.
- **Package's `DistilledCapacityRouter` is a strict behavioral subset**: only cheapest-within-tolerance hard-label CE + argmax routing + a caller-supplied `cost_fn`. It has no soft-target training, no blend-NLL training or eval, and no S1-style multi-arm label construction. Everything it does is covered by exactly one of the 10 script variants (mostly item 7's `fit_router`, generalized), but several script capabilities (S2's direct objective, S1's 5-arm factorial, t2/s1/h1's blend-NLL eval) have **no package equivalent at all** — this is a real, not merely renamed, capability gap, not a straightforward "one wins, delete the rest."

**Which one produced a certified result:**
`docs/width_mse_2026-07-16/verdict_variable_width_mse.md` — the width-MSE program's routing/selector claims (§6, "Payoff: the win is compute, not accuracy") are all read off `sinc_width_experiment.py`'s `_fit_selector_mse`/`_cheapest_within_tolerance_labels`/`_selector_eval_mse` machinery (item 6), run inside `kdropout_converged_width_experiment.py`. The package's `DistilledCapacityRouter` is **not** what produced that certified number — it is a later (Task F3) reimplementation of the same labeling rule wired into the package API, and its own test suite (`tests/test_phase3_dynamic_k.py`) validates it independently, not by reproducing the width-MSE verdict numbers.
Correction after verifying: `grep -rn "DistilledCapacityRouter" docs/` does NOT return nothing — it has real hits, all of them plan/spec documents rather than verdicts: `docs/plans/capacity_programme/flexnn-core.md` (the Task F3 spec that directed building the class, lines 110-136 read in full — its own "Non-goals" state "no in-training routing... no changes to selection" strands, i.e. this is a build spec, not a result), and `docs/probreg_benchmark/benchmark_spec.md`, which names the router as the intended k-selection mechanism but is explicitly self-labeled `**Status:** SPEC — frozen before any run` (line 6) and dated today (2026-07-20) — a not-yet-executed benchmark, not a certified number. So the directional claim holds (no doc certifies an experimental result produced BY `DistilledCapacityRouter`), but my first pass asserted "returned nothing" without having actually run the grep — that was a guess stated as fact and is corrected here.

**Migration risk if script routers were replaced by the package one:**
- Deleting `capacity_ladder_k6._RouterMLP`/`_train_router` would break `s1.py`, `s2.py`, `h1.py`, `sinc_width_experiment.py` (all import it directly) and transitively `hetero_width_experiment.py` (imports sinc_width). These are the scripts that produced the certified width-MSE verdict; their PREREGISTRATION/verdict docs cite exact function names, so replacing the implementation (even with an equivalent one) would break the paper trail unless the docs are also updated.
- The blend-NLL evaluation protocol (t2/s1/s2/h1) has no package equivalent, so any consolidation onto `DistilledCapacityRouter` would need to add it or explicitly declare those scripts' NLL-based comparisons out of scope for the package.
- `capacity_ladder_s2.py`'s direct-objective trainer is genuinely novel machinery not ported anywhere; if it's still an open research question (I did not find a verdict doc citing S2 as certified — only S1's "soft" winner is cited as certified per s2.py's own docstring, "S1 ... The winner was `soft`") it should not be silently dropped.

## Deliverable 4 — the API surface

Three model classes expose the "capacity dial" concept with three different parameter shapes — verified by reading each `__init__`/`predict`/`fit_router` signature directly, not inferred from names:

| Concept | `FlexibleHiddenLayersNN` (depth) | `FlexibleWidthNN` (width) | `ProbabilisticRegressionModel` (n_classes) |
|---|---|---|---|
| Capacity specification | `max_hidden_layers: int` — single upper bound, dial is implicitly `1..max_hidden_layers` | `widths: tuple[int,...]` — **explicit enumerated set**, e.g. `(16,32,48,64)`, NOT a range | `n_classes`, `n_classes_inf`, `max_n_classes_for_probabilistic_path` — **three separate params** for one conceptual dial |
| In-training selection enum | `layer_selection_method: LayerSelectionMethod` — `{NONE, SOFT_GATING, GUMBEL_SOFTMAX, STE, REINFORCE, NESTED}` (6 members) | `width_selection_method: WidthSelectionMethod` — `{NONE, DISTILLED}` **only** — no in-training strategies exist for width at all (verified: no `strategy_map`/`self.strategy` dispatch anywhere in `flexible_width_network.py`, unlike `flexible_neural_network.py:80`'s `self.strategy = strategy_map[self.layer_selection_method](self)`) | `n_classes_selection_method: NClassesSelectionMethod` — `{NONE, SOFT_GATING, GUMBEL_SOFTMAX, STE, REINFORCE, NESTED}` (6 members, same shape as depth's) |
| Regularizer enum | `depth_regularization: DepthRegularization` — `{NONE, DEPTH_PENALTY, ELBO, COST_AWARE_ELBO}` (4 members) | **none** — no width-regularization enum exists at all | `n_classes_regularization: NClassesRegularization` — `{NONE, K_PENALTY, ELBO}` (3 members) |
| `predict()` mode switch | `inference_mode: str = "soft"` (flexible_neural_network.py:386) | `inference_mode: str = "fixed"` (flexible_width_network.py:192) | `inference_mode: str = "soft"` (probabilistic_regression.py:598) |
| Post-hoc router | `fit_router()` (flexible_neural_network.py:437) → `DistilledCapacityRouter` | `fit_router()` (flexible_width_network.py:239) → `DistilledCapacityRouter` | `fit_router()` (probabilistic_regression.py:754) → `DistilledCapacityRouter` |

**Closed sets stored as raw strings, not enums (repo convention violation, verified in the code under audit itself):** `inference_mode` is typed `str` in all three `predict()` signatures — none of them use an `Enum`/`StrEnum`/`Literal`, and there is no `InferenceMode` type anywhere in `automl_package/enums.py` (grepped, zero hits). Each class validates it with an ad hoc `if inference_mode not in (...)` check (e.g. `flexible_width_network.py:204-205`) rather than the type system. This directly contradicts this repo's own stated convention ("Closed sets of variants get a type ... never scatter magic-string comparisons," per this session's operating rules) — worth flagging as in-scope for whoever designs the unified API, since it's the exact violation that convention exists to prevent. Additionally, the three classes' `inference_mode` closed sets are **not the same set of strings** and **not even the same DEFAULT semantics** (`"soft"` = differentiable full forward for depth/n_classes vs `"fixed"` = one explicit width for width) — a caller who has learned one class's `predict()` contract cannot safely guess another's.

**Silent defaults worth surfacing:** `FlexibleWidthNN.predict(..., width=None)` silently defaults to `max(self.widths)` (flexible_width_network.py:212) when `inference_mode="fixed"` and no width is given — a caller who forgets the `width=` kwarg gets the most expensive configured width with no warning, not an error.

## Deliverable 5 — what CANNOT simply be merged

1. **`depth_selection_toy._VectorRouterMLP` cannot be replaced by `capacity_ladder_k6._RouterMLP`** — verified in the source's own comment (depth_selection_toy.py:594-603): the k6 router hardcodes scalar input (`dims=[1, *hidden]`, `.reshape(-1,1)`) and the file states it confirmed this raises inside `nnf.cross_entropy` when given the one-hot WORD vector input this toy actually needs. Genuine shape incompatibility, not laziness.
2. **`capacity_ladder_t2.py`'s router/eval machinery cannot be silently replaced by the package's `DistilledCapacityRouter`** — it implements a blend-NLL evaluation protocol (`_blend_nll`, `_eval_router`) that the package router has no equivalent for at all; the package only does hard-argmax route + `cost_fn`-based deployed cost, never a probability-weighted blend score.
3. **`capacity_ladder_s2.py`'s direct-objective trainer (`_train_router_direct`) encodes an experiment-specific research question** ("what if the router is trained ON the deployment metric itself, with no derived label in between?" — s2.py:8-9) that has not been shown to be the certified/winning approach (S1's `soft` CE-to-label arm is the one the docstring calls "the winner" among S1's factorial). Merging it into the package would be presenting an unresolved research arm as the shipped default.
4. **`examples/depth_composition_toy.py`'s `RecurrentComposer`/`UntiedPerStepComposer`/`TiedFlatComposer` and `depth_graded_toy.py`'s `RecurrentPerLengthHead` cannot be ported into `FlexibleHiddenLayersNN` as-is** — they solve a GROUP WORD-PROBLEM (sequence recurrence over one-hot generator words) to get a GD-learnable depth-hunger signal at all (the depth.md doc explains this was forced by a theorem: smooth 1-D regression targets are provably depth-hunger-incompatible with gradient descent, citing Malach & Shalev-Shwartz 2019 and Malach et al. 2021). FlexNN's actual production use case is feedforward tabular regression/classification, not sequence recurrence — the certified depth *architecture conclusion* (shared readout wins) may be relevant, but the certifying *toy* is not directly reusable machinery for FlexNN's real forward pass.
5. **`capacity_accounting.py` cannot move into `automl_package/models/` without resolving its circular import first** (Deliverable 1's cost-accounting finding) — its per-architecture branches (`_executed_flops_nested_width`, `_executed_flops_independent_width`, `_executed_flops_shared_readout_per_width_affine`) reference `nested_width_net.py` classes that have no package home; moving the accounting function alone would either need to drop those branches or drag the width architecture census (Deliverable 2) along with it.
6. **`ResidualCascadeNet`/`MatryoshkaWidthNet` cannot be safely deleted as "just superseded"** without checking whether `docs/plans/width_dial_2026-07-11/cascade_execution_plan_2026-07-11.md`'s pre-registration is still considered a live commitment — I did not read that plan doc in full this pass (budget), so I cannot certify it's safe to drop; flagging as needs-a-human-check rather than asserting either way.
