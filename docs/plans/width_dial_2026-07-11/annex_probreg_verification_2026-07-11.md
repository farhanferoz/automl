## Traces

### A — Dynamic-k training path: where do bin boundaries come from at loss time?

Boundaries are **precomputed once per candidate k** before training starts (`_fit_single`, `probabilistic_regression.py:504-506`, loop over `k_values = range(2, max_k+1)`), stored in `self.precomputed_class_boundaries: dict[k -> edges]`. They are **not** recomputed per step/epoch. What varies per step is *which k's boundaries get used for which sample*, driven by that sample's `selected_k_values` for that forward pass.

Forward → loss path, quoted:
1. `ProbabilisticRegressionNet.forward` (`probabilistic_regression_net.py:151-159`) calls `self.n_classes_strategy.forward(x_input, n_classes_predictor_logits, boundaries=boundaries)`, returning `final_predictions, classifier_logits_out, selected_k_values, log_prob_for_reinforce, per_head_outputs`.
2. Inside the strategy (e.g. `_weighted_average_logic`, `base_selection_strategy.py:88-113`): `selected_k_indices = torch.argmax(mode_selection_probs, dim=1)` (95) → `selected_k_values = torch.where(selected_k_indices == last_mode, SENTINEL, selected_k_indices + 2)` (96-100) — one k value **per sample**, from that sample's own gate output.
3. `_calculate_custom_loss` (`probabilistic_regression.py:227-244`): `probabilistic_indices = torch.where(selected_k_values != DIRECT_REGRESSION_K_SENTINEL)[0]` (227) → `unique_k = torch.unique(k_values_prob)` (235) → `for k in unique_k: ... boundaries = self.precomputed_class_boundaries[k_int]; _, y_binned_k = create_bins(data=y_true_prob[mask].cpu().numpy(), unique_bin_edges=boundaries)` (237-242) → `classification_loss = masked_cross_entropy_loss(logits_prob, y_binned_prob, k_values_prob)` (244).

So the CE target for a given sample is built from the **precomputed, k-specific** (and, per claim 6, non-nested) boundary set for whatever k that sample's gate selected in that forward pass.

### B — Does the sampled k vary during dynamic-k TRAINING, and does binning vary with it?

Yes to both. `selected_k_values` is recomputed every forward call from `torch.argmax(mode_selection_probs, dim=1)` (`base_selection_strategy.py:95,129`), and `mode_selection_probs` comes from `n_classes_predictor(x_input)` (`probabilistic_regression_net.py:153`) — a function of the current batch's `x` and the gate's current (changing, trained) weights. So different samples in the same batch, and the same sample across epochs as the gate trains, can select different k. Each distinct k present in a batch pulls a different `self.precomputed_class_boundaries[k]` in the loop quoted in trace A (`probabilistic_regression.py:237-242`), so yes — the CE binning target varies with the selected k, sample-by-sample, batch-by-batch.

### C — `examples/capacity_ladder_h1.py`: which path, and does it inherit the rebinning defect?

It uses the **shipped ProbReg dynamic-k path directly** (`ProbabilisticRegressionNet`/`ProbabilisticRegressionModel`), not the `_capacity_ladder_nested.py` K4 surrogate. Evidence: `from automl_package.models.probabilistic_regression import ProbabilisticRegressionModel` (`capacity_ladder_h1.py:78`); arms (a)/(c) call `ProbabilisticRegressionModel(...).fit(...)` (`_fit_shipping_joint:242-251`, `_fit_fixed_k:259-267`); arm (b) builds the same net via `_build_phase1_host` (`:284-302`) and drives `net.classifier_layers`/`net.regression_module`/`net.direct_regression_head` directly. `_capacity_ladder_nested` (the K4 surrogate) is never imported in this file.

However, **all arms configure `optimization_strategy=ProbabilisticRegressionOptimizationStrategy.REGRESSION_ONLY`** (`NET_KW`, `capacity_ladder_h1.py:99`). In `_calculate_custom_loss`, `is_ce_active = self.optimization_strategy not in (REGRESSION_ONLY, GRADIENT_STOP)` (`probabilistic_regression.py:219-222`) is therefore **False** for every H1 arm that calls `.fit()`/`_calculate_custom_loss` — the classification-cross-entropy branch (and hence the per-k `create_bins`/boundary lookup from trace A/B) never fires. Arm (b)'s custom phase-1/phase-2 loop doesn't call `_calculate_custom_loss` or `create_bins` for CE at all — its `_phase1_masked_prefix_nll` (`:305-313`) uses `_per_k_log_density`, a pure Gaussian-mixture NLL over the SAME classifier logits/regression heads at every prefix length k, no rebinning. Its own docstring says so explicitly: `_seed_head_centroids` (`:214-224`) states "`precomputed_class_boundaries`/CE bookkeeping is intentionally NOT replicated: `optimization_strategy=REGRESSION_ONLY` ... never reads it."

**Conclusion: H1's recipe of record does NOT inherit the per-k-rebinning defect**, because every arm is configured with `REGRESSION_ONLY`, which structurally disables the CE/boundary-lookup code path where that defect lives. Component identity across k in H1 is instead governed by the single shared classifier's masked-prefix softmax (claim 8's mechanism), consistent with arm (c)'s fixed-k models each having their own single boundary set (only one `k` per instantiation, so no cross-k nesting question arises there either).

### D — `examples/_capacity_ladder_nested.py` (K4 surrogate): confirm stable component identity, independent heads, prefix-masked softmax

**Component identity stable across k** — one `weight_net` and one `component_heads` list are built once at `__init__` (`_capacity_ladder_nested.py:90-92`) and reused for every prefix length; `bin_centroids` only sets a one-time init bias (`:94-98`), it is not recomputed per k or per step. `masked_prefix_nll` (`:129-154`) computes `logits`/`mu`/`log_var` ONCE per call from the *same* component modules, then only changes the **mask**:
```
logits = self.weight_logits(x)               # (N, k_max) — line 145
mu, log_var = self.component_params(x)        # (N, k_max) — line 146
...
active = arange < k_draw.unsqueeze(1)         # line 150
masked_logits = logits.masked_fill(~active, float("-inf"))   # line 151
```

**Heads independent** — `self.component_heads = nn.ModuleList([vem._MLP(input_dim, 2, hidden) for _ in range(self.k_max)])` (`:92`); each of the `k_max` heads is its own MLP mapping `x -> (mu, log_var)` with no shared trunk (note: unlike ProbReg's `SeparateHeadsRegressionModule`, each K4 head takes raw `x` as input, not a probability scalar — a different design from claim 3's ProbReg heads).

**Prefix-masked softmax** — `log_w = masked_logits - torch.logsumexp(masked_logits, dim=1, keepdim=True)` (`:152`, a log-softmax renormalized over only the active/unmasked prefix) then `recon = torch.logsumexp(log_w + log_phi, dim=1)` (`:153`) — exactly a masked-softmax mixture NLL over the drawn prefix, matching the claimed "prefix-masked softmax" pattern.
